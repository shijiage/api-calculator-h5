## 1. 产品概述

### 1.1 一句话定义

**API 中转站个人计算器**是一款面向 AI API 个人用户的微信小程序，帮助用户在多个 API 中转站之间进行成本对比，通过 A/B 双站点计算生成可量化的性价比报告，辅助用户做出更经济的选站决策。

### 1.2 产品愿景

成为 AI API 个人用户的**首选成本决策工具**，让每一分钱花在最值得的站点上。

### 1.3 核心价值主张

| 价值维度 | 说明 |
|----------|------|
| **算得清** | 不同站点充值比例不同，工具自动换算为统一的「每 1K Tokens 人民币均价」，消除比较障碍 |
| **记得住** | 每次对比结果自动生成报告并保存，支持历史回看，不再依赖聊天记录和截图 |
| **比得出** | 可视化报告直观展示胜出站点、节省比例、节省金额，附带数据可信度评分 |
| **选得准** | 推荐站点目录提供市场参考，帮助发现更多可选站点 |

## 2. 产品背景与市场分析

### 2.1 行业背景

随着 ChatGPT、Claude、Gemini 等大语言模型的普及，大量个人用户通过「API 中转站」（第三方代理）来使用这些 API。中转站市场呈现以下特征：

- **站点众多**：市场上存在数十个中转站，各自定价策略不同
- **充值体系不统一**：有的站点 ¥7 = $1，有的 ¥7.5 = $1，有的按自定义比例
- **价格不透明**：同一模型在不同站点的实际人民币成本差异可达 20%–50%
- **用户决策困难**：缺乏标准化的比较工具，用户只能凭感觉或手动计算

### 2.2 竞品分析

| 维度 | 手动计算 (Excel/计算器) | 社群推荐 | 本产品 |
|------|------------------------|----------|--------|
| 使用门槛 | 高（需自建公式） | 低（被动接收） | 低（填数即出报告） |
| 数据持久化 | 需自行管理 | 无 | 自动云端同步 |
| 可视化 | 需自行制图 | 无 | 自动生成可视化报告 |
| 历史追溯 | 困难 | 不可能 | 支持历史回看与恢复 |
| 客观性 | 取决于个人 | 受推荐者利益影响 | 基于用户真实数据计算 |

### 2.3 市场机会

- 目前市场上**没有**专门针对 API 中转站成本对比的工具型小程序
- 用户痛点明确且高频（每次充值/换站都会触发比较需求）
- 微信小程序形态轻量，无需安装，适合工具类产品

## 3. 目标用户与用户画像

### 3.1 目标用户定义

**核心用户群：** 使用 API 中转站调用大语言模型的个人用户

**用户特征：**
- 有一定技术背景（开发者、AI 爱好者、内容创作者）
- 同时使用或考虑使用 2 个及以上中转站

- 月均 API 消费 ¥50–¥500

## 4. 用户场景与用户旅程

### 4.1 核心使用场景

#### 场景一：首次对比（新用户）

> 小李刚注册了两个中转站 A 和 B，各充了 ¥100。他想知道哪个站更划算。

1. 打开小程序，看到「30 秒上手」引导弹窗
2. 在站点 A 卡片填入充值金额 ¥100、获得额度 $14.28
3. 添加一条使用记录：输入 Tokens 5000、输出 Tokens 2000、消耗额度 $0.15
4. 在站点 B 卡片填入充值金额 ¥100、获得额度 $13.33
5. 添加一条使用记录：输入 Tokens 5000、输出 Tokens 2000、消耗额度 $0.18
6. 点击「生成性价比对比报告」
7. 查看报告：站点 A 每 1K Tokens ¥0.0015，站点 B ¥0.0020，A 站更划算，节省 25%

#### 场景二：历史复盘（老用户）

> 阿华上个月做过一次对比，这个月想看看结论是否还成立。

1. 打开小程序，进入「记录」页
2. 找到上个月的报告，点击查看详情
3. 点击「恢复对比配置」，回到计算页
4. 更新最新的使用数据
5. 重新生成报告，与上次对比

#### 场景三：分享报告

> 大明在群里讨论站点选择，想分享自己的对比结果。

1. 打开已生成的报告
2. 点击微信分享按钮
3. 报告以小程序卡片形式发送到群聊
4. 群友点击卡片可直接查看报告

#### 场景四：浏览推荐站点

> 小李想了解市场上还有哪些站点可以选择。

1. 进入「推荐」页
2. 按模型分类浏览推荐站点
3. 搜索特定站点名称
4. 查看站点的价格、延迟、在线率等指标
5. 复制站点链接去注册

### 4.3 用户漏斗

产品内置了完整的用户漏斗追踪：

```
进入计算页 (enter_calc)
    ↓

    ↓

    ↓

    ↓

**用户分群定义：**

| 分群 | 条件 |
|------|------|
| 浏览用户 | 仅访问过，未登录 |
| 登录用户 | 已登录，未生成报告 |
| 生成报告用户 | 至少生成过 1 次报告 |
| 复访用户 | 访问次数 ≥ 2 |

**北极星指标：** 周活跃报告用户数（过去 7 天内至少生成过 1 次报告的用户）

## 5. 产品目标与成功指标

### 5.1 产品目标

| 目标层级 | 目标 | 衡量方式 |
|----------|------|----------|
| **业务目标** | 成为 API 中转站用户的标准成本对比工具 | 用户量、报告生成量 |
| **用户目标** | 帮助用户每月节省 10%–30% 的 API 支出 | 报告中的节省金额 |
| **体验目标** | 30 秒内完成首次对比 | 首次报告生成时长 |

### 5.2 核心指标 (KPI)

| 指标 | 定义 | 目标值 (MVP) |
|------|------|-------------|
| **WAU** | 周活跃用户数 | 500+ |
| **周报告用户数** (北极星) | 过去 7 天生成过报告的独立用户数 | 200+ |
| **报告生成率** | 生成报告用户 / 进入计算页用户 | ≥ 40% |
| **次日留存率** | 次日再次打开的用户比例 | ≥ 15% |
| **7 日留存率** | 7 日内再次打开的用户比例 | ≥ 8% |
| **平均报告次数** | 每用户月均生成报告次数 | ≥ 2 |
| **分享率** | 分享报告用户 / 生成报告用户 | ≥ 10% |

### 5.3 数据埋点方案

产品采用**纯客户端本地埋点**方案，追踪以下数据：

| 数据项 | 存储键 | 说明 |
|--------|--------|------|
| 首次访问时间 | `calc_ana_first_seen_at` | 用户生命周期起点 |
| 最后访问时间 | `calc_ana_last_seen_at` | 活跃度判断 |
| 访问次数 | `calc_ana_visit_count` | 留存判断 |
| 登录次数 | `calc_ana_login_count` | 转化漏斗 |
| 报告生成次数 | `calc_ana_report_count` | 核心行为 |
| 报告生成日期 | `calc_ana_report_generated_days` | 北极星指标计算（保留 45 天） |
| 漏斗事件计数 | `calc_ana_funnel_counters` | 6 个漏斗事件各自的累计次数 |

**同步策略：**
```

**状态指示：**
- `idle`：初始状态，不显示
- `syncing`：显示「云端同步中…」
- `ok`：同步成功，不显示
- `failed`：显示红色警告条 + 重试按钮

**报告状态处理：**

| 对比状态 | Hero 标题 | 节省百分比 | 说明 |
|----------|-----------|-----------|------|
| `empty` | 请添加使用记录 | — | 两站均无数据 |
| `partial` | 请完善站点 X 数据 | — | 仅一站有数据 |
| `tie` | 两站成本相同 | 0% | 均价一致 |
| `a_win` | 站点 A 更具性价比 | X.X% | A 站均价更低 |
| `b_win` | 站点 B 更具性价比 | X.X% | B 站均价更低 |

**可信度评分维度：**

| 维度 | 1 分 | 2 分 | 3 分 | 4 分 | 5 分 |
|------|------|------|------|------|------|
| 样本量 | < 2 条 | 2–5 条 | 6–9 条 | 10–19 条 | ≥ 20 条 |
| 覆盖量 | < 2K Tokens | 2K–10K | 10K–50K | 50K–200K | ≥ 200K |
| 均衡度 | 比值 < 0.3 | 0.3–0.5 | 0.5–0.7 | 0.7–0.9 | ≥ 0.9 |
| 差异度 | < 2% | 2%–5% | 5%–10% | 10%–20% | ≥ 20% |

## 9. 核心算法说明

### 9.1 成本计算公式

```
① 充值比例 (rate)
   rate = 充值金额(¥) ÷ 获得额度($)
   · 获得额度为 0 时，按 1 处理（避免除零）

② 单条记录人民币花费
   recordCny = 消耗额度($) × rate

③ 总人民币花费
   totalCny = Σ(各有效记录的 recordCny)
            = Σ(消耗额度$) × rate

④ 每 1K Tokens 均价（核心对比指标）
   per1kCny = totalCny ÷ totalTokens × 1000
   · totalTokens = Σ(inputTokens + outputTokens)
   · totalTokens 为 0 时，per1kCny = 0

⑤ 对比结论
   · per1kCny 更低者为优
   · 节省百分比 = (高价 - 低价) ÷ 高价 × 100%
   · 节省金额 = (高价 - 低价) ÷ 1000 × 高价站的 totalTokens
   · 差值 < 1e-9 视为平局
```

### 9.2 有效记录判定

仅 `消耗额度($) > 0` 的记录参与计算，与 Web 端逻辑保持一致。

### 9.3 数据可信度评分算法

4 个维度独立评分，每维 1–5 分：

| 维度 | 计算方式 | 评分阈值 |
|------|----------|----------|
| 样本量 | A 记录数 + B 记录数 | 1/2/6/10/20 |
| 覆盖量 | A Tokens + B Tokens | 2K/10K/50K/200K |
| 均衡度 | min(A记录数, B记录数) / max(A记录数, B记录数) | 0.3/0.5/0.7/0.9 |
| 差异度 | 两站 per1kCny 差异百分比 | 2%/5%/10%/20% |

## 15. 功能专题：社区站点推荐 (V1.5)

### 15.1 功能概述

允许已登录用户分享自己用过的 API 中转站，形成 UGC 内容层，与官方维护的推荐数据相互补充。

**核心价值：**
- 用户：发现真实用户验证过的好站点，减少踩坑成本
- 产品：持续沉淀社区数据，增强内容壁垒

---

### 15.2 产品决策记录

| 决策项 | 结论 |
|--------|------|
| **推荐内容** | 站点名称 + 站点链接 + 一句话评价（三项必填） |
| **入口一** | 报告页 — 一键推荐当前对比站点 |
| **入口二** | 发现页 — 手动填写推荐 |
| **展示方案** | 方案 A：双位展示 — ①发现页独立「社区推荐」分区 ②同名站点卡片下方展示用户评价 |
| **质量门控** | 必须登录才能发布推荐 |
| **内容审核** | MVP 阶段无自动审核，依赖举报机制 |

---

### 15.3 用户流程

#### 流程一：从报告一键推荐

```
报告页
  └─ 点击某个站点的「推荐这个站点」按钮
       └─ 弹出填写弹窗（站点名已预填）
            ├─ 填写站点链接
            ├─ 填写一句话评价（≤50字）
            └─ 提交
                 ├─ 未登录 → 提示登录
                 └─ 已登录 → 调用云函数 → 成功提示
```

#### 流程二：发现页手动推荐

```
发现页 → 推荐站点 Tab
  └─ 点击右上角「+ 推荐站点」按钮
       └─ 弹出填写弹窗（三项均需手动填写）
            └─ 提交（同上）
```

#### 展示流程（方案 A）

```
发现页 → 推荐站点 Tab
  ├─ 官方推荐分区（现有数据）
  └─ 社区推荐分区（UGC 数据）
       └─ 展示所有用户提交的推荐卡片
            ├─ 站点名
            ├─ 链接（可复制）
            ├─ 一句话评价
            ├─ 提交者昵称（匿名显示前3位openid）
            └─ 发布时间

同时：
推荐站点 Tab → 官方站点卡片
  └─ 若社区中有同名站点，卡片底部展示「用户评价」折叠区
       └─ 显示最近 3 条用户评价
```
